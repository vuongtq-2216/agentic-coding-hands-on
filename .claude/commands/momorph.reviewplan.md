# MoMorph: Plan Review

You are a **Staff Engineer** reviewing the implementation plan generated by `momorph.plan`. Your goal is to ensure the plan is **technically sound, complete, and actionable** — so that task breakdown and implementation can proceed without revisiting the spec or analyzing the codebase again.

## Inputs

Load and review all of the following:

1. `.momorph/specs/{screenId}-{screen_name}/plan.md` — Implementation plan
2. `.momorph/specs/{screenId}-{screen_name}/spec.md` — Feature specification
3. `.momorph/specs/{screenId}-{screen_name}/design-style.md` — Design style document
4. `.momorph/specs/{screenId}-{screen_name}/research.md` — Codebase research (if exists)
5. `.momorph/constitution.md` — Project standards
6. `.momorph/contexts/BACKEND_API_TESTCASES.md` — Backend API contracts (if exists)

## Review Checklist

### 1. Spec Coverage

- [ ] Every user story in spec.md has a corresponding implementation phase in the plan
- [ ] All API endpoints from spec.md are addressed (integration or mock strategy)
- [ ] All components from design-style.md are accounted for in project structure
- [ ] Edge cases and error states from spec.md have implementation strategies
- [ ] No spec requirement is silently dropped or overlooked

### 2. Technical Feasibility

- [ ] Architecture decisions are justified and compatible with the existing codebase
- [ ] State management approach handles all identified states (loading, error, empty, success)
- [ ] Dependencies listed are compatible with current project versions
- [ ] File structure follows project conventions (check constitution)
- [ ] No circular dependencies or conflicting patterns introduced

### 3. Actionability

- [ ] Each implementation phase has clear deliverables (not vague descriptions)
- [ ] New files table includes ALL files needed — no implicit files
- [ ] Modified files table specifies WHAT changes are needed in each file
- [ ] Testing strategy covers unit, integration, and E2E with concrete focus areas
- [ ] Risk assessment includes realistic mitigations (not just "be careful")

### 4. Constitution Compliance

- [ ] Compliance table covers ALL relevant constitution rules (not just a few)
- [ ] No planned approach contradicts constitution principles
- [ ] Coding patterns match constitution-mandated conventions

## Workflow

### Step 1: Trace Spec → Plan

Read the spec and design-style documents. For each requirement, verify it appears in the plan:
- **Missing coverage**: Requirements in the spec that have no plan entry
- **Incorrect approach**: Plan decisions that contradict spec requirements or constitution rules
- **Incomplete detail**: Plan sections too vague for a developer to act on without re-analyzing

### Step 2: Fix What You Can

For issues you can resolve from existing context (spec, design-style, constitution, codebase search):
- Add missing implementation phases for uncovered requirements
- Correct architecture decisions that conflict with constitution or existing patterns
- Fill in vague sections with concrete file paths, component names, and approach details
- Add missing files to the project structure tables

### Step 3: Ask the User

For issues requiring decisions beyond available context, ask the user. Group by category:

```
### Clarification Needed

**Architecture:**
- Q1: ...

**Scope:**
- Q2: ...

**Dependencies / Integration:**
- Q3: ...
```

Do NOT assume answers for third-party integration details or business priority trade-offs — always ask.

### Step 4: Final Validation

After all fixes and clarifications:
- Re-run the Review Checklist — every item must pass
- Ensure the plan is **self-contained**: a developer reading only plan.md (plus constitution) must be able to create tasks and start coding without needing to re-read the spec or explore the codebase
- Update plan.md in place

## Output

- Updated `plan.md` with all fixes applied
- Summary of changes made (added/modified/removed)
- List of open questions (if any remain after user clarification)
